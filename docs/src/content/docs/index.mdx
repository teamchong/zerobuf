## Why zerobuf

Every WASM tool today copies data across the JS/WASM boundary. wasm-bindgen, AssemblyScript, Emscripten — all serialize in, copy, deserialize out. Two full copies per call.

zerobuf eliminates this. JS and WASM share one `WebAssembly.Memory`. Both sides agree on a binary layout. Proxy objects on the JS side read/write directly to WASM linear memory.

## Quick Example

```typescript
import { zerobuf } from "zerobuf";

const buf = zerobuf(wasm.exports.memory);

// Create object directly in WASM linear memory
const point = buf.create({ x: 1.0, y: 2.0, label: "origin" });

// Proxy reads/writes WASM memory — no copy
point.x = 3.14;

// Pass pointer to WASM function
transform(point.__zerobuf_ptr);

// WASM wrote result in-place — JS reads it back, still zero copy
console.log(point.x);
```
